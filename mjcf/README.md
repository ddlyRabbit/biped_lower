# MuJoCo Sim2Sim

MuJoCo simulation environment for the biped robot. Used for sim-to-sim validation
of Isaac Lab trained policies and PD standing tests.

## Architecture

### MJCF Model (`sim2sim/robot.mjcf`)

Generated from `urdf/light/robot.urdf` using K-Scale's `urdf2mjcf` tool.

- **Source URDF:** light variant (16.6kg, +X forward)
- **Conversion:** `urdf2mjcf` with patched `iter_meshes` for multi-visual-geom URDFs
- **Collision:** Mesh-based (urdf2mjcf duplicates each mesh as visual + collision)
  - Robot geoms: `contype=1, conaffinity=0` (collide with floor only, no self-collision)
  - Ground plane: `contype=1, conaffinity=15`
  - `<contact><exclude>` pairs for adjacent bodies (shin↔ankle↔foot)
- **Actuators:** 12 `<motor>` actuators (torque mode, PD computed in Python)
- **Sensors:** Per-joint pos/vel/force + orientation quaternion + angular velocity
- **Joints:** `damping=0.01, armature=0.01, frictionloss=0.01` (from urdf2mjcf defaults)

### Standing Controller (`sim2sim/stand_mujoco.py`)

PD position control holding the robot at default joint positions.

**Default Pose (MuJoCo-tuned, matches Isaac training):**
```
R_hip_pitch: -0.08    L_hip_pitch:  0.08
R_hip_roll:   0.00    L_hip_roll:   0.00
R_hip_yaw:    0.00    L_hip_yaw:    0.00
R_knee:       0.25    L_knee:       0.25
R_foot_pitch: -0.17   L_foot_pitch: -0.17
R_foot_roll:  0.00    L_foot_roll:  0.00
```

**PD Gains (tuned for stable standing):**
```
Joint         Kp    Kd    Effort
hip_pitch     10    2.0      40   ← LOW Kp! Higher oscillates
hip_roll      30    3.0      40
hip_yaw       30    3.0      40
knee          40    3.0      60
foot_pitch    80    3.2      30   ← HIGH Kp — main load bearing
foot_roll     14    0.8      30
```

**Key insight:** Hip pitch Kp must stay ≤10 — the large rotational inertia
of the upper body causes PD instability at Kp≥15 (oscillation → saturation → fall).

### Policy Playback (`sim2sim/play_mujoco.py`)

Runs an ONNX student policy in the MuJoCo environment.

**Observation vector (45d):**
```
[0:3]   base angular velocity (gyro sensor)
[3:6]   projected gravity (quat_rotate_inverse)
[6:9]   velocity commands (vx, vy, wz)
[9:15]  hip positions (L_roll, R_roll, L_yaw, R_yaw, L_pitch, R_pitch) relative to default
[15:17] knee positions (L, R) relative to default
[17:19] foot pitch positions (L, R) relative to default
[19:21] foot roll positions (L, R) relative to default
[21:33] joint velocities (Isaac runtime order)
[33:45] previous actions
```

**Action → Torque pipeline:**
```
actions = onnx_model(observations)                    # 12 actions
actions_clipped = clip(actions, -10, 10)
targets = default_pos + actions * action_scale        # 0.5, ankle_roll=0.25
torques = Kp * (targets - current_pos) + Kd * (0 - vel)
torques = clip(torques, -effort_limits, effort_limits)
data.ctrl[:] = torques
```

**Joint ordering:**
- MuJoCo actuator order: R_hp, R_hr, R_hy, R_kn, R_fp, R_fr, L_hp, L_hr, L_hy, L_kn, L_fp, L_fr
- Isaac runtime order: L_hp, R_hp, L_hr, R_hr, L_hy, R_hy, L_kn, R_kn, L_fp, R_fp, L_fr, R_fr
- Mapping arrays `MJ_TO_ISAAC_IDX` / `ISAAC_TO_MJ_IDX` handle conversion

## Sim2Sim Gap

The Isaac-trained policy does not directly transfer to MuJoCo due to:

1. **Actuator model:** Isaac uses DelayedPDActuator with friction; MuJoCo uses ideal motor
2. **Contact dynamics:** Isaac PhysX vs MuJoCo's PGS solver
3. **Collision geometry:** Isaac convex decomposition vs MuJoCo mesh collision
4. **Inertia properties:** urdf2mjcf mass accumulation differs from Isaac URDF loader

The PD standing controller works perfectly (10s+ stable) because it doesn't rely
on learned dynamics. The policy transfer requires matching the sim dynamics or
training directly in MuJoCo (MJX).

## Training Config Reference (V68b)

The MJCF defaults match the Isaac training config:

- **Actuator:** DelayedPDActuator, min_delay=1, max_delay=3 (5-15ms)
- **Friction randomization:** (0.5, 1.25)
- **Self-collisions:** disabled
- **Air time reward:** Berkeley impact-based (contact sensor)
- **Symmetry:** data augmentation + mirror loss (coeff=1.0)
- **Joint deviations:** hip=-0.1, knee=-0.01, hip_pitch=-0.01, ankle_pitch=-0.01, ankle_roll=-0.1

## System Identification

Actuator characterization pipeline to match sim dynamics to real hardware.

### Scripts

```
deploy/scripts/
├── motor_sysid.py          ← Real robot recording
└── motor_sysid_isaac.py    ← Isaac Sim recording (copy to GCP)
```

### Pipeline

**Step 1: Real Robot Recording**
```bash
# On Pi — robot SUSPENDED, powered
cd ~/biped_lower
python deploy/scripts/motor_sysid.py --all

# Or single joint:
python deploy/scripts/motor_sysid.py --joint R_hip_pitch --kp 250 --kd 5
```

Records step response + sine sweeps (0.5/1/2/5/10Hz) per motor type:
- RS04 (hip_pitch, knee): Kp=250, Kd=5
- RS03 (hip_roll, hip_yaw): Kp=150, Kd=5
- RS02 (ankle): Kp=40, Kd=5

Single motor active per test for max CAN rate (~500-1000Hz).
Ankle joints use parallel linkage transform (commands in joint space,
recorded in joint space via inverse transform).

**Step 2: Isaac Sim Recording**
```bash
# On GCP
python motor_sysid_isaac.py --all --actuator delayed
python motor_sysid_isaac.py --all --actuator implicit  # for comparison
```

Same tests, gravity disabled, fixed base. 200Hz sim rate.
Output format matches real robot CSVs.

**Step 3: Compare & Fit**

Overlay real vs sim step/sine responses. Adjust Isaac actuator parameters
(stiffness, damping, friction, delay, armature) until responses match.

Key metrics:
- Step response: rise time, overshoot, settling time
- Sine tracking: gain and phase lag per frequency (Bode plot)
- Friction: hysteresis loop width
- Delay: command-response cross-correlation

## Deploy CAN Drivers

Three options for motor control:
- **Python sync** (`can_bus_node`): Original, 50Hz. Good for debugging.
- **Python async** (`can_bus_node_async`): Threaded, ~200Hz.
- **C++ (`can_bus_node_cpp`)**: Direct SocketCAN, ~300Hz. Recommended for deployment.

Select via launch param: `can_driver:=can_bus_node_cpp`
